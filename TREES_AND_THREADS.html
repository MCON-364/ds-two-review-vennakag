★Tree Traversals: BFS and the Three DFS

A traversal is a systematic way of visiting every node in a tree exactly once. We covered one BFS strategy and three DFS strategies. The DFS three are: In-order, Pre-order, and Post-order.

Working tree for all examples

5 / \ 3 7 / \ / \ 1 4 6 8

DFS Pre-order

Visit: Root → Left → Right

You record a node before you go deeper. Think of it as "stamp your foot when you first arrive at a node."

Algorithm (recursive):

visit(node)
preOrder(node.left)
preOrder(node.right)

Result on the example tree:

5 → 3 → 1 → 4 → 7 → 6 → 8

Used for: copying a tree, prefix expression evaluation, serialising a tree.

DFS In-order

Visit: Left → Root → Right

You record a node between its left and right subtrees. For a BST this produces sorted output — this is the traversal happening inside every TreeMap / TreeSet iteration.

Algorithm (recursive):

inOrder(node.left)
visit(node)
inOrder(node.right)

Result on the example tree:

1 → 3 → 4 → 5 → 6 → 7 → 8

Used for: sorted output from a BST, expression trees (infix), validating a BST.

DFS Post-order

Visit: Left → Right → Root

You record a node after both its children. Think "process the children before you process the parent."

Algorithm (recursive):

postOrder(node.left)
postOrder(node.right)
visit(node)

Result on the example tree:

1 → 4 → 3 → 6 → 8 → 7 → 5

Used for: deleting a tree (delete children before parent), postfix expression evaluation, computing subtree sizes.

BFS — Level-order

Visit: Level by level, left to right

Uses a queue (not recursion). You add the root, then repeatedly dequeue a node, visit it, and enqueue its children.

Algorithm (iterative with Queue):

Queue<Node> q = new LinkedList<>();
q.add(root);
while (!q.isEmpty()) {
  Node n = q.poll();
  visit(n);
  if (n.left  != null) q.add(n.left);
  if (n.right != null) q.add(n.right);
}

Result on the example tree:

5 → 3 → 7 → 1 → 4 → 6 → 8

Used for: shortest path in unweighted graphs, finding nodes at depth k, printing a tree level-by-level.

Side-by-side comparison

Traversal Type Data structure needed Order: root relative to children Result on example tree Key use case
Pre-order DFS Call stack (recursion) Root first 5 3 1 4 7 6 8 Copy / serialize a tree
In-order DFS Call stack (recursion) Root between 1 3 4 5 6 7 8 ← sorted! BST sorted output; TreeMap iteration
Post-order DFS Call stack (recursion) Root last 1 4 3 6 8 7 5 Delete a tree; evaluate expression
BFS / Level-order BFS Queue (iterative) Level by level 5 3 7 1 4 6 8 Shortest path; level-k nodes
The trick to remember pre/in/post:
The prefix tells you when you visit the root:
  • Pre-order → root first (before children)
  • In-order → root in the middle (between children)
  • Post-order → root last (after children)
Left always comes before Right.
Why in-order = sorted for a BST:
Because of the BST invariant — everything to the left is smaller. When we finish the entire left subtree before visiting the current node, and only then move to the right subtree, we visit nodes in ascending order. This is exactly what TreeMap and TreeSet do when you iterate them.
Quick check: For the tree above, trace BFS yourself with a Queue. Start by adding 5. What order do you dequeue? Does it match the table?
Quick check 2: If we insert keys 10, 5, 15, 3, 7 into a BST in that order, what does in-order traversal produce? What about pre-order?

1Binary Search Tree (BST)

A BST is a binary tree where every left child is smaller and every right child is larger. This condition makes ordered lookup, insertion, and deletion efficient.

The BST invariant

For every node N:

  • All values in the left subtree are < N
  • All values in the right subtree are > N

Example tree containing: 5, 3, 7, 1, 4, 6, 8

5 / \ 3 7 / \ / \ 1 4 6 8

BST operations

OperationAverage caseWorst case (skewed)
SearchO(log n)O(n)
InsertO(log n)O(n)
DeleteO(log n)O(n)
In-order traversalO(n)O(n)

In-order traversal = sorted output

Walking a BST in-order (left → root → right) visits every node from smallest to largest. This is why tree-backed collections can iterate in sorted order for free.

Key insight: An in-order traversal of the example tree above visits: 1, 3, 4, 5, 6, 7, 8. Sorted!

The BST weakness: skewed trees

If we insert keys in sorted order (1, 2, 3, 4, 5…), the tree degrades into a linked list:

1 \ 2 \ 3 \ 4 ← O(n) to find 4
Pitfall: A BST has no mechanism to re-balance itself. Worst-case performance is O(n), no better than a linked list. This is why Java does not expose a plain BST — it uses a self-balancing tree instead.
Think about it: Given the BST above with 5 at the root, what path do we follow to find the value 6? How many comparisons do we make?

2Red-Black Tree

Java's TreeMap and TreeSet are backed by a Red-Black tree — a self-balancing BST that guarantees O(log n) in all cases, even for sorted insertions.

The five Red-Black properties

  1. Every node is either red or black.
  2. The root is always black.
  3. Every null leaf (sentinel) is considered black.
  4. If a node is red, both of its children must be black (no two consecutive red nodes on any path).
  5. For every node, all simple paths from that node to its descendant leaves contain the same number of black nodes.
Why these rules work: Property 5 (equal black-height on every path) is the key. It ensures the longest path is at most twice the shortest path, so the tree is always roughly balanced. This guarantees O(log n) height.

Rotations and re-coloring

When an insertion or deletion violates a Red-Black property, the tree fixes itself using:

  • Rotations — restructure the tree locally (left-rotate or right-rotate) without changing BST ordering.
  • Re-coloring — flip node colors to restore the Red-Black properties.

These fixups take O(log n) time and happen automatically inside TreeMap and TreeSet — we never call them directly.

Red-Black tree guarantees

OperationGuaranteed time
Search / getO(log n)
Insert / putO(log n)
Delete / removeO(log n)
In-order traversalO(n)
First / last keyO(log n)
Floor / ceilingO(log n)
Bottom line: Because TreeMap and TreeSet use a Red-Black tree internally, all their operations are O(log n) regardless of the insertion order. The skewed-tree problem of a plain BST does not apply here.
Think about it: Why can't we just use property 5 alone (equal black-heights) without property 4 (no two consecutive red nodes)? What would go wrong?

3Java Tools: TreeMap and TreeSet

Java's tree-backed collections all implement NavigableMap or NavigableSet, which adds powerful range query operations on top of the basic sorted ordering.

TreeSet

Sorted, no duplicates

TreeSet<String> names = new TreeSet<>();
names.add("Charlie");
names.add("Alice");
names.add("Bob");

// Iterates: Alice, Bob, Charlie
for (String name : names) { ... }

names.first();    // "Alice"
names.last();     // "Charlie"
names.floor("B"); // "Bob" (greatest <= "B")
names.ceiling("B"); // "Bob" (smallest >= "B")
names.headSet("Bob"); // [Alice] (strictly before Bob)
names.subSet("Alice", true, "Charlie", false); // [Alice, Bob]
TreeMap

Sorted key-value map

TreeMap<String, Integer> scores = new TreeMap<>();
scores.put("Charlie", 88);
scores.put("Alice", 95);
scores.put("Bob", 72);

// Iterates in key order: Alice, Bob, Charlie
scores.entrySet().forEach(e -> ...);

scores.firstKey();         // "Alice"
scores.lastEntry();        // Charlie=88
scores.floorKey("B");      // "Bob"
scores.headMap("Charlie"); // {Alice=95, Bob=72}
scores.subMap("Alice", true, "Charlie", false);
Custom Order

Comparator-based ordering

// Sort by salary descending
TreeSet<Employee> byPaycheck = new TreeSet<>(
    Comparator.comparingInt(Employee::salary).reversed()
);

// Sort map keys by string length then alphabetically
TreeMap<String, Integer> byLength = new TreeMap<>(
    Comparator.comparingInt(String::length)
              .thenComparing(Comparator.naturalOrder())
);

TreeMap + Streams: the natural combination

Because TreeMap iterates in sorted key order, streaming its entry set gives us sorted data without any extra sort step.

// Build a frequency map from a list of words — sorted alphabetically for free
TreeMap<String, Long> frequency = words.stream()
    .collect(Collectors.groupingBy(w -> w, TreeMap::new, Collectors.counting()));

// Top 5 words by count (frequency map → stream → sort by value → limit)
List<String> top5 = frequency.entrySet().stream()
    .sorted(Map.Entry.comparingByValue(Comparator.reverseOrder()))
    .limit(5)
    .map(Map.Entry::getKey)
    .collect(Collectors.toList());

// Group employees by department — departments in alphabetical order, employees sorted within each group
TreeMap<String, TreeSet<Employee>> roster = employees.stream()
    .collect(Collectors.groupingBy(
        Employee::department,
        TreeMap::new,
        Collectors.toCollection(TreeSet::new)
    ));
Key pattern: Passing TreeMap::new as the map factory to Collectors.groupingBy is the standard way to get a sorted grouping from a stream.

4NavigableMap: Range Queries

NavigableMap (implemented by TreeMap and ConcurrentSkipListMap) is the interface that makes tree-backed maps so powerful for range queries.

MethodWhat it returnsExample with keys: A, B, C, D, E
headMap(toKey) Keys strictly less than toKey headMap("C") → {A, B}
headMap(toKey, inclusive) Keys up to toKey, inclusive flag controls boundary headMap("C", true) → {A, B, C}
tailMap(fromKey) Keys greater than or equal to fromKey tailMap("C") → {C, D, E}
subMap(from, to) Keys in [from, to) — from inclusive, to exclusive subMap("B", "D") → {B, C}
subMap(from, fi, to, ti) Keys with explicit inclusive/exclusive flags on both ends subMap("B", true, "D", true) → {B, C, D}
floorKey(key) Greatest key ≤ the given key floorKey("C") → "C"
ceilingKey(key) Smallest key ≥ the given key ceilingKey("C") → "C"
descendingMap() A reverse-order view of the entire map Iterates E → D → C → B → A
Important: The views returned by headMap, tailMap, and subMap are live views — they reflect any changes to the underlying map. If you only need a snapshot, collect the values into a list immediately.

Range queries in practice

TreeMap<String, Book> titleIndex = buildTitleIndex(); // keyed by title

// All books whose title starts with "A" (prefix range query)
String prefix = "A";
NavigableMap<String, Book> aBooks =
    titleIndex.subMap(prefix, true, prefix + Character.MAX_VALUE, false);

// Stream the matching books
List<Book> result = new ArrayList<>(aBooks.values()); // snapshot

5Concurrent Sorted Collections

When multiple threads need sorted access simultaneously, TreeMap and TreeSet are not thread-safe. Java provides two drop-in concurrent replacements.

ConcurrentSkipListMap

Thread-safe replacement for TreeMap.

  • Keys always in sorted order
  • All NavigableMap methods available
  • O(log n) get, put, remove
  • Lock-free for most reads
ConcurrentSkipListMap<Long, String> log =
    new ConcurrentSkipListMap<>();

// Multiple threads can safely call:
log.put(key, message);
log.tailMap(timestamp).values();
ConcurrentSkipListSet

Thread-safe replacement for TreeSet.

  • Elements always in sorted order
  • All NavigableSet methods available
  • O(log n) add, contains, remove
  • Iteration reflects concurrent modifications
ConcurrentSkipListSet<ScoreEntry> board =
    new ConcurrentSkipListSet<>();

// Multiple threads can safely call:
board.add(new ScoreEntry(player, score, ts));
board.stream().limit(10)
     .collect(Collectors.toList()); // top 10

Why a Skip List instead of a Red-Black tree?

Red-Black trees require complex tree rotations during updates, which are hard to make lock-free. A Skip List achieves O(log n) sorted operations using simpler pointer links that can be updated atomically, making concurrent access much more practical.

Common mistake: Wrapping a TreeMap in Collections.synchronizedMap does make individual method calls atomic, but iteration is still not safe — the entire loop must be synchronized externally. ConcurrentSkipListMap has no such restriction.

Full concurrent pattern: sorted log with ExecutorService

ConcurrentSkipListMap<Long, String> eventLog = new ConcurrentSkipListMap<>();
AtomicLong sequence = new AtomicLong(0);

// Thread-safe insertion with unique composite key
void logEvent(long timestamp, String message) {
    long key = timestamp * 1_000_000L + sequence.getAndIncrement();
    eventLog.put(key, message);
}

// ↓ eventsEach and sources are PARAMETERS of this method ↓
void runConcurrentSources(List<String> sources, int eventsEach) {

    ExecutorService pool = Executors.newFixedThreadPool(sources.size());

    for (String source : sources) {      // 'source' — new copy each iteration

        pool.submit(() -> {              // lambda CAPTURES 'source' and 'eventsEach'
                                         // from the enclosing method scope
            for (int i = 0; i < eventsEach; i++)   // ← comes from the method parameter above
                logEvent(System.currentTimeMillis(), source + "-" + i);
        });
    }
    pool.shutdown();
    pool.awaitTermination(30, TimeUnit.SECONDS);
}

// After all threads finish, stream the sorted results:
List<String> recent = eventLog.descendingMap().values().stream()
    .limit(10)
    .collect(Collectors.toUnmodifiableList());
Where does eventsEach come from?
It is a parameter of the enclosing method runConcurrentSources. The lambda inside pool.submit forms a closure — it can read variables from the surrounding method scope as long as they are effectively final (never reassigned after the lambda is created).

Both captured variables satisfy that rule:
• eventsEach — method parameter, never changed → effectively final ✓
• source — loop variable, a brand-new copy each iteration → effectively final ✓
What breaks effective-final?
If you wrote eventsEach++ anywhere in the method, the compiler would refuse to compile the lambda with: "Variable used in lambda expression should be final or effectively final." The same applies to source: you cannot do source = source.trim() before using it inside the lambda.

How this connects to the final

The final can change the story, but the shape of the problem will still be recognizable.

If the final says... We should think... Likely tools
"Results must be in alphabetical / sorted order" Tree-backed collection; no sort step needed on iteration. TreeMap, TreeSet, Collectors.toCollection(TreeSet::new)
"Find all items between X and Y" Range query — NavigableMap or NavigableSet. subMap, headMap, tailMap
"Group by some field, keep each group sorted" Two-level tree: TreeMap for the groups, TreeSet inside each group. Collectors.groupingBy(..., TreeMap::new, Collectors.toCollection(TreeSet::new))
"Multiple threads need sorted access simultaneously" Tree-backed collection is not thread-safe — switch to the concurrent version. ConcurrentSkipListMap, ConcurrentSkipListSet
"Count / average / min / max over a collection" Stream summary statistics or groupingBy with counting. summaryStatistics(), Collectors.counting(), Collectors.summarizingDouble
"Multiple threads submit work, query results later" Producer writes concurrently; consumer reads after pool shuts down. ExecutorService, ConcurrentSkipListMap, AtomicLong
Final reminder: Trees give you sorted order and O(log n) operations. Streams let you express what you want without loops. Concurrent collections let you share sorted state safely across threads. When all three come together, you have solved the whole problem.
